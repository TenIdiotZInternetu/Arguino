\section{Simulator module}

\begin{figure}[htbp]
    \centering
    \includegraphics[width=\textwidth,keepaspectratio]{./img/uml/b_simulator.pdf}
    \caption{\texttt{simulator} module architecture}
    \label{fig:sim_module}
\end{figure}

The \texttt{Simulator} class is initialized by the \texttt{main.cpp}, and its pointer is stored as a static field of the class itself, making it a singleton. It owns both the \texttt{ArduinoState} and \texttt{EventQueue} objects and provides access to them through its static classes. Namely \texttt{handle\_event} is called by the \texttt{Arduino.cpp}, passing events to the queue and consuming all events that are already waiting. 

The events are created with static factory classes of the \texttt{Event} struct. In these, lambda functions are being created, utilizing the \texttt{Simulator}'s \texttt{state()} method, which provides direct access to the \texttt{ArduinoState}.

\subsection{\texttt{ArduinoState} class}

The class holds the digital states and modes of pins in \texttt{std::array}s. Sizes of these arrays match the specification of the \texttt{Arduino UNO R3 SMD} board. The analog pins also stored in this class are not utilized at the point of writing this thesis. 
The pins can be read and modified with setters and getters analagous to those in \texttt{Arduino.h}, but instead of integers, the paramters are of types defined in \texttt{SimulatorTypes.hpp}. Namely \texttt{digital\_t} is \texttt{bool}, \texttt{analog\_t} is \texttt{double}, \texttt{pin\_t} is \texttt{uint8\_t} and \texttt{PinMode} is an enum attaining values \texttt{In} and \texttt{Out}.

The parameterless overloads return reference to an entire array. This was particularly useful when processing snapshots, before events were implemented. The setter methods return bool indicating success of the operation. Currently all methods always indicate success, but to achieve higher quality of life for users, warnings and errors should be logged instead of crashing the simulator immediately, e.g. when accessing pins outside of array's range.

The \texttt{\_timer} owned by this class represents the simulation time. The \texttt{Timer} class is not described in detail in this thesis as it was originally designed for a different project, and a lot of its features are not interesing for this particular work. Under the hood it uses \texttt{std::chrono} library. The \texttt{deltaTime()} method returns microseconds passed from the timer's last step. The timers of this thesis are stepped exclusively at its intialization.

\subsection{\texttt{Simulator} singleton class}

The singleton's instance is created on the heap within the static \texttt{init} method and its pointer stored in the \texttt{s\_instance} member. The \texttt{eventCallback} paramater of this method must be set to the handler, that should be called upon event consumption. In our implementation that is the \texttt{on\_event} function from \texttt{ipcAdapter.hpp}. This callback is stored into \texttt{f\_eventCallback}.

\texttt{state} and \texttt{queue} methods are used to access the objects owned by the \texttt{Simulator}. But the queue is more typically accessed through the \texttt{handle\_event} and \texttt{handle\_events} calls. They both consume all remote events waiting in the queue. The \texttt{handle\_event} variant also puts a local event in the queue and consumes after all remote events with lower LVT.

The \texttt{Simulator} also owns a \texttt{Timer}, this one representing the real time passed. At the moment, both timers show roughly the same time, but it is anticipated, that the simulation timer could be manipulated by the user in the future versions of the project. Lastly, the \texttt{Simulator} class holds a shared pointer to a logger. It also exposes shortcut methods for passing log messages of different verbosity to the logger. This particular logger is used for any messages unrelated to the IPC.

\subsubsection{\texttt{handle\_event} method} \label{sssec:handle-event}

\begin{lstlisting}[language=csharp, caption={\texttt{Simulator::handle\_event} method}]
void $Simulator$::#handle_event#($Event$ event)
{
    auto& queue = s_instance->_eventQueue;
    queue.#enqueue_local#(event);

    do {
        queue.#execute_next_event#();
    } while (event.#id#() != queue.#last_executed_event#().#id#());

    s_instance->#f_eventCallback#(event);
}
\end{lstlisting}

The event is enqueued to the local event queue, and all remote events with lower LVT are consumed. To check that the local event has been consumed, we had to add a unique identifier member to the \texttt{Event} struct distinguishing different event instances. This \texttt{id} is compared to the \texttt{id} of the previously consumed event, and the while loop is stopped when these two match. We couldn't use the LVT value as, even though it is unique for local events, it doesn't have to be unique across processes. In some frontend implementations, remote events can actually have higher LVT than the local ones.

Due to the semantics of LVT, we ensure that every remote event that occured before the local event has been consumed. The exception are events that already occured remotely, but haven't been fast enough to reach the backend's event queue. This can occur with TCP connection, or with very frequent \texttt{Arduino.h} calls in the sketch. These events will most likely be still consumed in the next call of \texttt{handle\_event}. But for full correctness more synchronization techniques should be implemented, depending on the use case.

When the local event has finally been consumed, it is signalled to the IPC through the \texttt{f\_eventCallback}.

\subsection{\texttt{Event} struct}

The members of the \texttt{Event} struct mostly match the event specification in the subsection \ref{subsec:events}. The additions are the \texttt{id} and \texttt{action} members. We already justified the use \texttt{id} member in the previous section. It is assigned in the constructor from the static \texttt{s\_nextEventId}, which is then incremented. The \texttt{action} is assigned as a lambda function from within the static factory methods. This lambda is executed during the \texttt{EventQueue}'s \texttt{execute\_next\_event} call.

\subsubsection{Event factories}

\begin{lstlisting}[language=csharp, caption={\texttt{Event::write} function}]
$Event$ $Event$::#write#($pin_t$ pin, $digital_t$ value)
{
    $Event$ event($Type$::Write, pin, value);
    event.action = [=]() {
        $Simulator$::#state#().#set_digital#(pin, value);
        $Simulator$::#log_info#($std$::#format#(
            "Written value {} to pin {}", value, pin));
    };
    return event;
}
\end{lstlisting}

The above is an example on how the factory methods should look like. The constructor takes a type and variadic arguments corresponding to that type. the constructor is statically asserted to not accept more than 3 variadic arguments. The action is defined as a lambda, with call to the \texttt{Simulator}'s \texttt{state} getter, modifying its values. The outcome is then logged through the \texttt{Simulator}'s logger. 

At this point, the LVT of the event is not known. That is the responsibility of the scheduling entity. For local events it is the \texttt{EventQueue}, for remote events it is IPC, which has to decode LVT from an incoming message.

\subsection{\texttt{EventQueue} class}

The event queue is in reality two queues. One storing local and one storing remote events. As we only have two threads running concurrently in the backend, this simplifies the synchronization significantly. We also completely avoid locking the structure in order to prevent race conditions. These queues are populated with the public \texttt{enqueue\_local} and \texttt{enqueue\_remote} methods.

\xxx{nextLvt, processedEvents}

\subsubsection{\texttt{execute\_next\_event} method}

\begin{lstlisting}[language=csharp, caption={\texttt{EventQueue::execute\_next\_event} method}]
void $EventQueue$::#execute_next_event#()
{
    auto& queueAhead = 
        (#earliest_queue_lvt#(_localEvents) < 
         #earliest_queue_lvt#(_remoteEvents))
         ? _localEvents
         : _remoteEvents;

    $Event$ processedEvent = queueAhead.#front#();
    queueAhead.#pop#();

    processedEvent.#action#();
    _nextLvt = $std$::#max#(
        _nextLvt, processedEvent.localVirtualTime + 1);
    _processedEvents.#push#(processedEvent);
}
\end{lstlisting}

Whichever event that has the minimal LVT of both queues has to be executed next. That is to preserve the semantics of LVT, such that every event with higher LVT is the consequence of an event with lower LVT. We expect that the queues are sorted by the LVT of their events, from lowest to highest. For local events that is given by sequential execution of the sketch. For remote events this is given by the IPC technologies we've chosen. TCP guarantees that data are received in the same order as they were sent. In the shared memory approach, we deliberately write and store messages in a sequential order. So to determine the next event, all we have to do is peek at the first element of both of the queues, and select the one with lower LVT. That is ultimately how \texttt{queueAhead} is obtained.

The event is then popped from the queue. The \texttt{action} lambda stored within it is finally executed. The event can now be considered consumed. Then the \texttt{\_nextLvt} is decided. If the consumed event has the highest encountered LVT yet, we simply increment this value. If it does not, we don't change the value, as it would mean the subsequent events would technically occur in the past.

\subsection{\texttt{Arduino.cpp}}

\begin{lstlisting}[language=csharp, caption={\texttt{Arduino.cpp}}]
using namespace arguino::simulator;

void #digitalWrite#(uint8_t pin, uint8_t val)
{
    if ($Simulator$::#state#().#get_digital#(pin) == val) return;

    $Simulator$::#handle_event#(
        $Event$::#write#(pin, val == HIGH)
    );
}

int #digitalRead#(uint8_t pin)
{
    $Simulator$::#handle_events#();
    return $Simulator$::#state#().#get_digital#(pin);
}

unsigned long #millis#()
{
    return $std$::#floor#($Simulator$::#state#().#get_time#() / 1000);
}

void #pinMode#(uint8_t pin, uint8_t mode)
{
    $PinMode$ pinMode = mode == OUTPUT ? $PinMode$::Out : $PinMode$::In;
    if ($Simulator$::#state#().#get_pin_mode#(pin) == pinMode) return;

    $Simulator$::#handle_event#(
        $Event$::#set_pinmode#(pin, pinMode)
    );
}
\end{lstlisting}

At last, we implement the four functions specified in the goals of the thesis. \texttt{millis} simply takes the simulation time and divides it by 1000 to get the elapsed milliseconds. The \texttt{digitalWrite} and \texttt{pinMode} functions are implemented using the \texttt{handle\_event} call with the corresponding event factory passed as an argument. If the argument for these events doesn't differ from the current value stored in \texttt{ArduinoState}, we exit the function early. In sketches with these functions in the outermost \texttt{loop} scope, which is actually a common pattern, the condition can save millions of redundant events and log messages.

The \texttt{digitalRead} call consumes all waiting events, and then simply reads the pin of \texttt{ArduinoState}. Out of the four functions, this is the only one that can be problematic with regard to delayed event reception described in the \texttt{handle\_event} method (\ref{sssec:handle-event}), as it is the only one dependant on previous \texttt{Arduino.h} calls. It is possible for \texttt{digitalRead} to return a state outdated by a few microseconds. 

A proposed solution is to add an event serving as a kind of condition variable. This event would be sent to the frontend and the same event would be sent back after any necessary frontend changes would have been resolved. Until then, the backend's simulation thread would remain in a spin lock, pausing the simulation timer, handling remote events and checking for the expected synchronization event. Only then the function would return a value and the execution of the sketch would resume.

This proposal haven't been implemented at the time of writing the thesis due to time concerns. This incorrect behavior hasn't been yet observed with the provided tests, and it probably won't be noticed on simpler sketches of a target user audience.

\section{Executable module}

\begin{figure}[htbp]
    \centering
    \includegraphics[width=\textwidth,keepaspectratio]{./img/uml/b_executable.pdf}
    \caption{\texttt{executable} module architecture}
    \label{fig:exe_module}
\end{figure}

The \texttt{executable} module puts all the systems together and compiles into an executable file. \texttt{sketch.cpp} doesn't come with the source code. It has to be preprocessed from the sketch user wishes to simulate. \texttt{main.cpp} defines how the sketch is used, and orders the IPC adapter to use it in context of the used IPC mechanism.

Two execution targets are provided in the CMake configuration, one for each IPC mechanism. They differ in which IPC adapter implementation is used, and which program options are supported with that implementation. The also differ in whether the \texttt{tcp-server} or the \texttt{shared-memory} module is linked with the executable.

\subsection{\texttt{main.cpp}}

\begin{lstlisting}[language=csharp, caption={\texttt{main} function}]
int #main#(int argc, char** argv)
{
    options = #parse_arguments#(argc, argv);
    #init_simulator#(#on_event#);
    #run_simulator_with_ipc#(#simulator_loop#, options);
}
\end{lstlisting}

The \texttt{main} function of the C++ project parses the command line arguments, initializes the Simulator singleton, defines the simulation \texttt{loop} and let's the IPC adapter handle the rest. The \texttt{on\_event} function passed to \texttt{init\_simulator} is included from \texttt{ipcAdapter.hpp} and implemented by the adapter chosen via CMake execution target. Simulation logger and it's verbosity are created in the \texttt{init\_simulator} as well.

The \texttt{simulator\_loop} calls the the required functions included from \texttt{sketch.cpp}: the \texttt{setup} function once and the \texttt{loop} function in an infinite while loop. The \textit{Reboot} event is sent as the very first event of each simulation. This is useful when the frontend keeps running, but backend is being restarted or recompiled with a different sketch. The lack of such an event would preserve the frontend state of the old simulation, and it would get desynchronized very quickly.

The command line arguments are parsed using the \texttt{boost::program\_options} library. The values in their expected data types are stored in the \texttt{ProgramOptions} struct. An instance of this struct is stored statically in the \texttt{main.cpp} file, and passed down to the IPC adapter by reference.

\subsection{\texttt{ipcAdapter.hpp}}

The reposonsibility of the \texttt{run\_simulator\_with\_ipc} method is to initialize the classes used for IPC and store them in the implementing \texttt{.cpp} file. That typically involves creating a new thread for listening to messages incoming from the frontend process. This thread has to be joined at some point after the simulation loop. That is why we pass the loop function as an argument, to be invoked at the right time.

The \texttt{on\_event} function is passed to the \texttt{Simulator} and is called every time a local event is consumed. The event is encoded into a message using the function included from \texttt{encoder.hpp}. The message can be then transmitted over the IPC, which has already been initialized before the very first local event is created.

The \texttt{init\_ipc\_logger} function is already defined as it provides the IPC logging setup common for any adapter. We call this function at the beginning of each \texttt{run\_simulator\_with\_ipc} definition.

Implementations are located in the \texttt{ipc} directory of the \texttt{executable} module. But we should first understand how were the IPC mechanisms implemented, so we moved the adapter documentations to their respective chapters (subsections \ref{subsec:tcpadapter} and \ref{subsec:shmemadapter})

\subsection{\texttt{textEncoder.cpp}}

The \texttt{encoder.hpp} declares a function that encodes local events into messages, and decodes messages into remote events \texttt{decode\_event} outputs the result event as a reference parameter. Its return value indicates success of the operation. That is, whether or not was the received message in a valid a format. The \texttt{UNKNOWN\_EVENT} constant is used to identify undefined event types.
\footnote{This can occur when implementing new events, but forgetting to add new cases in encoding. This should never happen with the source code attached with this thesis.}


At the time of writing this thesis, \texttt{textEncoder.cpp} is the only encoder implementation. It handles human readable messages, the format of which is described in more detail in the section \ref{sec:msg_format}. The message is divided into 3 to 6 parts delimited by a colon, the first 2 of which are common for every event type. Hence the constant string formats for each of the types. \texttt{encode\_message} identifies the type, fills the event values into the respective format, and its done.

\subsubsection{Decoding a message}

\begin{lstlisting}[language=csharp, caption={\texttt{decode\_event} implementation}]
bool #decode_event#(const $std$::$string$& message, $Event$& event)
{
    $parts_array_t$ parts = #split#(message);
    $std$::$array$<int, MAX_ARGS> args;

    for (int i = 0; i < MAX_ARGS; ++i) {
        if (!#decode_int#(parts[COMMON_PARTS + i], args[i])) break;
    }

    if (parts[2] == "W") {
        event = $Event$::#write#(args[0], args[1]);
    }
    else if (parts[2] == "P") {
        event = $Event$::#set_pinmode#(args[0], ($PinMode$)args[1]);
    }
    else {
        return false;
    }
    if (!#decode_int#(parts[0], event.timestampMicros) || 
        !#decode_int#(parts[1], event.localVirtualTime)) {
        return false;
    }
    return true;
}
\end{lstlisting}

\texttt{decode\_message} has to manually split the message into substrings and parse each substring individually. First we decode arguments in a loop. then we decode the type encoding character, and based on that we call the appropriate factory method with the right arguments. In the end we decode the timestamps from the common parts.

At each of these parts we can encounter a value that doesn't conform to its expected data type or a known event type, and the decoding fails. Then we have to return \texttt{false}. At this point, the event can be already partially initialized, but it is the responsibility of the caller to discard this instance, and fall into some fail state.

\section{TCP Server module}

IPC over TCP is implemented using the \texttt{boost::asio} library. It serves as an abstraction over asynchronous I/O and networking resources of the OS. The core class of the library is the \texttt{io\_context} type, in which the network operations are scheduled and executed. The \texttt{tcp::acceptor} type attaches to the context and listens for any TCP connection on the given endpoint (ip and port). At that time \texttt{ConnectionHandler} instance is created. This happens within the \texttt{TcpServer} class.

Once a connection has been established, it connects to a socket owned by the \texttt{ConnectionHandler}. Inside its \texttt{handle} method, the connection handler waits asynchrnously for any incoming data. Once data are available, and the message delimeter is found within them, the data is passed to the \texttt{handle\_message} where it is preprocessed and the passed to a handler callback passed to the class from the TCP adapter.

If a message needs to be sent over TCP, it is first passed to \texttt{TcpServer} and that passes it to the \texttt{ConnectionHandler}. It is put to an outbox queue and sent asynchronously over the same socket, within the same \texttt{io\_context}.
\footnote{The general outline of the server and connection handler is taken from \texttt{boost::asio} tutorials \cite{asio_async_server}. A CppCon presentation by Michael Caisse \cite{caisse_asio} was incredibly helpful for practical understanding of the library and common patterns. }

\subsection{\texttt{TcpServer} class}

The class initializes the \texttt{io\_context}, the \texttt{acceptor} and the \texttt{endpoint} within its constructor. One of the contructor's arguments is callback, that has to be called once a message is received by the \texttt{ConnectionHandler}. We expect only one client to connect at a time. \footnotemark We store a shared pointer to this handler and pass it the \texttt{\_messageHandler} callback once it is ready. Local messages that need to be sent to the frontend are also passed to the same \texttt{ConnectionHandler}. If there is no connection, sending a message blocks the calling thread, hence the \texttt{\_connectionCv} condition variable.

\footnotetext{A more robust architecture would of course allow multiple clients to be connected at once, which would require to decouple handler from the server. But only one connection also means we don't need any threadpool to handle the traffic. }

\subsubsection{Accepting connections}

\begin{lstlisting}[language=csharp, caption={\texttt{launch} method}]
void $TcpServer$::#launch#()
{
    _acceptor.#listen#();
    #start_accepting#();
    _ioContext.#run#();
}
\end{lstlisting}

The server starts to listen for connections by calling the \texttt{launch} method. Within it the acceptor starts to listen for connection with the \texttt{listen} method. What must happen after a connection is established is defined by the \texttt{start\_accepting} method. \texttt{\_ioContext.run()} blocks the calling thread and starts executing any asynchronous operations that have been scheduled for execution. The method return once there is no work to be done, so we have to make sure there is at least one scheduled operation.

\begin{lstlisting}[language=csharp, caption={\texttt{start\_accepting} method}]
void $TcpServer$::#start_accepting#()
{
    auto newConnection = handler_t::#create#(
        _ioContext, _messageHandler, _logger);

    _acceptor.#async_accept#(
        newConnection->#socket#(), [=, this](auto error) {
            if (!error) {
                _logger->#log_info#("Connection accepted!");
                _connectionHandler = newConnection;
                _connectionCv.#notify_one#();
                _connectionHandler->#handle#();
            }
            else {
                _logger->#log_error#("...");
            }
            #start_accepting#();
    });
}
\end{lstlisting}

Async TCP operations generally require a reference to a \texttt{tcp::socket} object and a callback to call once the operation has been executed by the \texttt{io\_context}. A socket is always associated with a single connection, so we define it in our \texttt{ConnectionHandler} class and create its instance beforehand.

In the callback we notify the \texttt{\_connectionCv} that a connection has been found, store the pointer of the handler and call its \texttt{handle} method. Then we must call the \texttt{start\_accepting} again. This is another common pattern among async operations. It makes sure that another async operation is scheduled immediately after the first one has been completed, listening for new connections, messages, etc. Without them \texttt{io\_context} simply stops. This looks like a recursive pattern, but remember that the outer function has finished long before the callback is invoked by the executing context.

\subsection{\texttt{ConnectionHandler} class}

The \texttt{ConnectionHandler} owns the socket, used for asynchronous TCP operations. It provides the public \texttt{handle} and \texttt{post\_message} methods for scheduling reading and writing operations respectively. The private \texttt{handle\_message} and \texttt{write\_from\_queue} are used within the callbacks of these operations.

The \texttt{ConnectionHandler} inherits the \texttt{enable\_shared\_from\_this} with itself as the template argument. This allows us to call the inherited \texttt{shared\_from\_this}, that creates a new shared pointer, pointing to itself. Those are used to prolong the lifetime of the object with every subsequent scheduled operation, as each one of the creates a new pointer. This pattern is very common in ASIO servers, especially when the handlers are not owned by any higher entity.

\subsubsection{Receiving messages}

\begin{lstlisting}[language=csharp, caption={\texttt{handle} method}]
void $ConnectionHandler$::#handle#()
{
    _isConnected = true;

    $boost$::$asio$::#async_read_until#(
        _socket,
        $boost$::$asio$::#dynamic_buffer#(_incomingBuffer),
        MESSAGE_DELIMITER,
        [me = this->#shared_from_this#()]
        (auto error, size_t messageSize) {
            if (error == $boost$::$asio$::$error$::eof) {
                me->#disconnect#();
            }
            else {
                me->#handle_message#(error, messageSize);
                me->#handle#();
            }
        }
    );
}
\end{lstlisting}

Upon new connection, the handler starts immediately listening for new messages with the \texttt{async\_read\_until} function. Aside from usual arguements it requires a delimiting character (semicolon in our case) and an \texttt{asio::buffer}. Buffers in the context of asio are views of contiguous regions of memory. We use a dynamic buffer (meaning it can change size) and back it with the \texttt{\_incomingBuffer} string.

The method reads data from the network until the delimiting character has been encountered. Then the portion of the data until the delimiter is copied into the buffer. And after that the callback is called with the number of copied character passed as an argument. We process the data in the buffer with \texttt{handle\_message} method.

\begin{lstlisting}[language=csharp, caption={\texttt{handle\_message} method}]
void $ConnectionHandler$::#handle_message#(
    $boost$::$system$::$error_code$ error, size_t messageSize)
{
    if (error) {
        _logger->#log_error#("...");
        return;
    }

    if (_incomingBuffer.#empty#()) return;

    const $std$::$string$ message = 
        _incomingBuffer.#substr#(0, messageSize - 1);
    _logger->#log_debug#("...");
    #_messageHandler#(message);

    _incomingBuffer.#erase#(0, messageSize);
}
\end{lstlisting}

We check for errors and strip the message of its delimiter. Finally we pass the message to the \texttt{\_messageHandler} which was bestowed to us from the \texttt{tcpHandler.cpp} through the server. 

\subsubsection{Posting messages}

\begin{lstlisting}[language=csharp, caption={\texttt{post\_message} method}]
void $ConnectionHandler$::#post_message#(const $std$::$string$& message)
{
    {
        $std$::$unique_lock$ lock(_queueMutex);
        _queueCv.#wait#(lock, [this]() {
            return _outboxQueue.#size#() <= MAX_QUEUE_SIZE;
        });

        _outboxQueue.#push#(message);
    }

    $boost$::$asio$::#post#(_executor, 
        [me = this->#shared_from_this#()] {
            me->#write_from_queue#();
        });
}
\end{lstlisting}

For scheduling messages to send we use a queue of strings called \texttt{\_outboxQueue}. Since this method is most likely going to be called from a thread different that one of \texttt{io\_context} we need to wrap the sending logic in \texttt{post} function. This schedules the \texttt{write\_from\_queue} method in the execution context of the \texttt{io\_context} thread. The \texttt{\_executor} argument represents this. It has been retrieved from \texttt{io\_context} in the connection handler's construtor.

We set the maximum count of messages in the queue, so that the simulator doesn't do unnecessary work if it isn't listened to. Once this count has been reached, the calling thread is blocked with the \texttt{\_queueCv} condition variable.

\begin{lstlisting}[language=csharp, caption={\texttt{write\_from\_queue} method}]
void $ConnectionHandler$::#write_from_queue#()
{
    if (_isWriting) return;
    if (_outboxQueue.#empty#()) return;

    _isWriting = true;

    $boost$::$asio$::#async_write#(_socket,
        $boost$::$asio$::#buffer#(_outboxQueue.#front#() + ';'),
        [me = this->#shared_from_this#()]
        (auto error, size_t bytes_written) {
            if (error) {
                me->_logger->#log_error#("...");
            }
            else {
                me->_logger->#log_debug#("...");
                me->_outboxQueue.#pop#();
                me->_queueCv.#notify_one#();
            }

            me->_isWriting = false;
            me->#write_from_queue#();
        }
    );
}
\end{lstlisting}

The messages are sent with the \texttt{async\_write} operation. It requires a buffer from which the data will be copied to the network. We back this up by one element of the \texttt{\_outboxQueue} with appended delimiter. Once succeeded, the message is removed from the queue and the condition variable is notified with the change of the queue's size. 

It is possible for the \texttt{write\_from\_queue} method to be invoked twice before \texttt{async\_write} executes even once. This would end up by sending the same message multiple times. So we guard the method with \texttt{\_isWriting} bool, that is set to true once the method is called, and only set to false after the \texttt{async\_write} has been executed. This causes that the number of \texttt{post\_message} calls doesn't match the number of \texttt{async\_write} calls. So we call the method repeatedly from the callback, such that we send all the messages waiting in the queue.

\subsection{\texttt{tcpAdapter.cpp}} \label{subsec:tcpadapter}

\begin{lstlisting}[language=csharp, caption={\texttt{on\_received\_tcp\_message} function}]
static void #on_received_tcp_message#(const $std$::$string$ message)
{
    $Event$ event;
    if (#decode_event#(message, event)) {
        $Simulator$::#queue#().#enqueue_remote#(event);
    }
    else {
        IpcLogger->#log_error#("...");
    }
}
\end{lstlisting}

The implementation of \texttt{tcpAdapter.cpp} is quite simple. The \texttt{TcpServer} is instantiated with \texttt{on\_receive\_tcp\_message} as its message handler, and it's \texttt{launch} method is invoked in a new thread. Then simulation loop starts.

\texttt{on\_event} and \texttt{on\_receive\_tcp\_message} are semantically opposite to each other. The former encodes an event into message and enqueues it to the server. The latter decodes a message into an event in enqueues it to the \texttt{Simulator}.

\section{Shared Memory module}

\begin{figure}[htbp]
    \centering
    \includegraphics[width=\textwidth,keepaspectratio]{./img/uml/b_sharedmemory.pdf}
    \caption{\texttt{shared-memory} module architecture}
    \label{fig:shmem_module}
\end{figure}

The \texttt{shmemAdapter.cpp} holds a \texttt{TwoWayBuffer} object. It represents the two circular buffers, one producing and one consuming. It creates the OS-level memory mapped object using the \texttt{boost::interprocess} library, and splits it into two regions of the same size, one for each buffer. 

The \texttt{CircularBuffer} class implements the data structure of the buffer itself. It was designed deliberately for use with shared memory. It holds the \texttt{MemoryRegion} instance, which abstracts away the low level details of accessing the shared memory directly. We defined a forward iterator for the \texttt{CircularBuffer} to simplify the manupilation of its actual data.

\subsection{\texttt{MemoryRegion} class}

This class provides an abstraction over the \texttt{interprocess::mapped\_region} type. The aforementioned type isolates a region of memory within the \texttt{interprocess::shared\_memory\_object} and provides direct access to it. We create the \texttt{mapped\_region} object in the constructor, defined by its size and the offset from the starting address of the \texttt{shared\_memory\_object} in bytes. We set its permission to read and write.

The methods of \texttt{MemoryRegion} consist mostly of transformations from \texttt{void*} to \texttt{uint8\_t*}, and conversions between pointers and offsets. We cannot perform pointer arithmetic on \texttt{void*}, so it is useful to have the pointers already in form that allows passing through memory byte by byte. We already mentioned the importance of offsets in the section \ref{subsec:ipc_shmem}.

The templated \texttt{at} method uses reinterpretation casting to automatically retrieve the underlying type from raw bytes, and returns a reference to it.

\subsection{\texttt{CircularBuffer} class} \label{subsec:b_circular_buffer}

The data structure occupies the entire space defined by the underlying \texttt{MappedRegion}. The producer and consumer offsets occupy the first 16 bytes of the region. The rest is filled with the raw data, simply referred to as \texttt{buffer} in the code. In the context of this structure, the offset must be interpreted as the distance from the address from the start of the buffer.

To simplify this notion we defined a forward iterator with this offset as the underlying iteration unit. It dereferences into \texttt{uint8\_t}. Incrementing over the boundary of the memory region sets the offset back to 0. \texttt{producer\_it} and \texttt{consumer\_it} turn the offsets stored in the raw memory into these iterators. They can be written back with the \texttt{write\_producer} and \texttt{write\_consumer} methods.

For the purpose of writing and reading messages (or bigger structures in general), we define the \texttt{write} an \texttt{consume\_until} methods. These implicitly access the data at \texttt{producer\_it}, respectively \texttt{consumer\_it}, onwards. \texttt{write} comes in two variants, one for contiguous ranges and one for any other templated type. The return value indicates whether or not does the buffer have enough available space to insert the type. \texttt{consume\_until} takes a delimeter as the point at which to stop reading the data. It returns the read bytes in an \texttt{std::vector} and moves the producer offset. 

\subsubsection{Writing to buffer}

\begin{lstlisting}[language=csharp, caption={\texttt{CircularBuffer::write} method}]
template <$std$::$ranges$::$contiguous_range$ TRange>
inline bool $CircularBuffer$::#write#(const TRange& data)
{
    auto bytes = $std$::#as_bytes#($std$::$span$(data));

    // We don't want the buffer to be fully filled
    if (#bytes_available#() <= bytes.#size#()) { 
        return false; 
    }

    size_t bytesUntilBufferEnd = #end#() - #producer_it#();

    if (bytes.#size#() < bytesUntilBufferEnd) {
        $std$::#memcpy#(
            &*#producer_it#(), bytes.#data#(), bytes.#size#());
    }
    else { // Message written on a boundary
        auto firstBytes = bytes.#first#(bytesUntilBufferEnd);
        auto lastBytes = bytes.#last#(
            bytes.#size#() - bytesUntilBufferEnd);

        $std$::#memcpy#(
            &*#producer_it#(), firstBytes.#data#(), firstBytes.#size#());
        $std$::#memcpy#(
            &*#begin#(), lastBytes.#data#(), lastBytes.#size#());
    }

    #write_producer#(#producer_it#() + bytes.#size#());
    return true;
}
\end{lstlisting}

The range variant of \texttt{write} is the one that does all the hard work. The other one just just transforms the type \texttt{T} into an \texttt{std::span} with the constant reference to \texttt{T} as it's only element. This qualifies it as a range.

First we turn the range into a span of bytes, so we can access each one of them using pointer arithmetics. Then we check if there is enough avilable space to store all the bytes. We have to make sure that there is always at least one byte unused. Otherwise would the consumer and producer offsets equal. We specifically interpret this state as the buffer being empty. Without this specification it would be ambiguous whether is the buffer empty or full.

Then we check if there is enough space left until the end of the buffer. If not, we will have to write the data on its boundary by splitting it in two. The \texttt{firstBytes} get copied to the producer offset, and the rest at the offset 0.

\subsubsection{Reading from buffer}

\begin{lstlisting}[language=csharp, caption={\texttt{CircularBuffer::consume\_until} method}]
$std$::$vector$<uint8_t> $CircularBuffer$::#consume_until#(
    uint8_t delimeter)
{
    $iterator_t$ producerIt = #producer_it#();
    $iterator_t$ consumerIt = #consumer_it#();

    auto delimeterIt = $std$::#find#(
        consumerIt, producerIt, delimeter);
    bool delimeterNotFound = delimeterIt == producerIt;
    if (delimeterNotFound) return {};

    // +1 to include the delimeter
    size_t dataLength = delimeterIt - consumerIt + 1;
    size_t wrapsAround = dataLength > #end#() - consumerIt;
    $std$::$vector$<uint8_t> result(dataLength);

    if (!wrapsAround) {
        $std$::#memcpy#(&result[0], &*consumerIt, dataLength);
    }
    else { // message lies on the boundary
        size_t firstBytesLength = #end#() - consumerIt;
        size_t lastBytesLength = delimeterIt - #begin#();

        $std$::#memcpy#(&result[0], &*consumerIt, firstBytesLength);
        $std$::#memcpy#(
            &result[firstBytesLength], &*#begin#(), lastBytesLength);
    }

    #write_consumer#(delimeterIt + 1);
    return result;
}
\end{lstlisting}

First we need to copy the producer offset as it can change during the execution of the function. Since \texttt{iterator\_t} is a fully qualified forward iterator, we can pass the offset iterators as arguments of \texttt{std::find} in order to find the iterator of the delimeter. If the delimeter hasn't been found, we return an empty vector, indicating that the operation doesn't make sense with the current state of the buffer. 

Otherwise we proceed analogously to \texttt{write}. We determine the length of the message, and if it could fit in until the end of the buffer. If not, it lies on the boundary, so we have to copy portion of the data from the end of the buffer, and portion from its beginning.

\subsection{\texttt{TwoWayBuffer} class}

\texttt{TwoWayBuffer} is a wrapper around the \texttt{CircularBuffer} class. It owns the instances of both buffers, and limits access to them. \texttt{write} passes its arguments to \texttt{write} method of the producer buffer and \texttt{consume\_until} passes its argument to the method of the same name of the consumer buffer.

The class also creates and owns the \texttt{shared\_memory\_object}. This object either creates a new OS-level shared memory object, or opens one with the given name if it already exists. Opening the object means it doesn't need to be destroyed, and the frontend doesn't need to shift into a panic state. At the construction of the circular buffers, both of the offsets are reset 0, effectively clearing any previous data.

The memory object needs to be specified its size, which is always a multiple of the page size on the given OS (usually 4096B). \texttt{TwoWayBuffer} then dedicates half the size of the memory object to each of the circular buffers.

\subsection{\texttt{shmemAdapter.cpp}} \label{subsec:shmemadapter}

\begin{lstlisting}[language=csharp, caption={\texttt{run\_shmem\_consumer} function}]

static void #run_shmem_consumer#()
{
    while (true) {
        while (!_shmem->#consumer#().#is_empty#()) {
            $std$::$vector$<uint8_t> data = _shmem->#consumer#()
                .#consume_until#(
                    MESSAGE_DELIMETER);

            if (data.#empty#()) {
                IpcLogger->#log_error#("...");
                continue;
            }

            $std$::$string$ message(data.#begin#(), data.#end#());
            IpcLogger->#log_debug#("...");

            $Event$ event;
            if (#decode_event#(message, event)) {
                $Simulator$::#queue#().#enqueue_remote#(event);
            }
            else {
                IpcLogger->#log_error#("...");
            }
        }
    }
}
\end{lstlisting}

The adapter utilizes the shared memory through the high level \texttt{TwoWayBuffer} wrapper. It provides it its name and size in pages from options. Then it creates a single thread that infinitely checks for new messages in the consumer buffer. Instead of the spin lock (inner while loop in \texttt{run\_shmem\_consumer}) it would be more appropriate to use interprocess semaphore to check the contents of the buffer, saving some resources. Inside the loop a message delimeted by a semicolon is retrieved, decoded and enqueued. \texttt{on\_event} method is as simple as encoding the event and writing it to the buffer, given there is enough space.
