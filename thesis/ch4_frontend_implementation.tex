\subsubsection{Note on the \texttt{event} keyword}

Until now we've been talking about events exclusively in the context of IPC. But since C\# defines it's own \texttt{event} keyword, used multiple times in the frontend's code, using the two different meanings could become confusing. So for the duration of this chapter, we will write \textit{event} (in italics) every time we talk about the C\#'s keyword and the associated patterns. \section{Component Management project - Circuitry and Scenes} \label{sec:cm_project_circuitry_and_scenes}

Classes of the \texttt{Circuitry} and \texttt{Scenes} directories are a tightly coupled system, encapsulating the core logic of the frontend. The \texttt{Scene} classes comprises of nothing more than a list of components and a list of electrical nodes. Components of the same electrical node interact with each other through the interface of pins. These can either listen to changes of digital state on the node, or they could assert a high state to it (driving).

\texttt{Component} is the class overridden by user in order to create a custom component. It holds \texttt{ComponentConfiguration} structs statically in a dictionary. This struct contains the metadata of a \texttt{Component} subtype, available sprites and pin identifiers. Each \texttt{Component} instance owns a set of \texttt{Pin} instances orchestrating the circuit's logic. 

One of the \texttt{Component} subtypes included in this project is the \texttt{Arduino} class. It is the component that connects from IPC to the backend simulator. Since the component is so important and it depends so heavily on the \texttt{IIpcAdapter} (it is basically the frontend for the adapter) we decided to document it in the IPC Adapter project section (\ref{sec:f_ipcadapter}). 

Components, sprites and nodes of a scene are created by the static \texttt{SceneFactory} class and its loader interfaces. These are documentated in detail in the section \ref{sec:f_factory}.

\subsection{\texttt{Pin} class}

\texttt{IsDriving} and \texttt{State} are the most important properties of the \texttt{Pin}. \texttt{State} is a read only getter property, that evaluates based on the state of the node it is connected to (if any). \texttt{IsDriving} can be changed by the owner \texttt{Component} by using either \texttt{SetHigh}, \texttt{SetLow} or \texttt{SetValue} methods. For read only pins, these methods trigger a warning, and the value isn't changed. If the value is different from before these methods were called, the \textit{DrivingChangedEvent} is invoked. This notifies the connected node to update its internal driving pins counter.

By default, the pin is not connected to any node. \texttt{ConnectToNode} method is called within the \texttt{ElectricalNode} to change that. In this method, the pin is added to the node's list of pins. The \texttt{NotifyStateChanged} method is called by the node, when its state changes. The method is also called right after the pin has been succesfully connected or disconnected.

The \texttt{Disconnect} method is implemented for cases, when \texttt{ConnectToNode} is called, while the pin is already connected to a different node. Or when such a request comes from the GUI. But at the time of writing this thesis, such mechanisms haven't been implemented yet. The pin is ever connected only once during the initialization of a node in the \texttt{SceneFactory}. These changes in connection are signaled by \textit{PinConnectedEvent} and \textit{PinDisconnectedEvent}.

If \texttt{NotifyStateChanged} is by any chance invoked on a write only pin, it triggers a warning, and the change isn't signalled to the component. The change is also not signalled when the pin is driving. In all other cases, pin's \textit{StateChangedEvent} is invoked. This \textit{event} is subscribed to by the parent \texttt{Component}, which handles it with its \texttt{OnPinStateChanged} virtual method.

The \texttt{PinPrototype} record struct is used to encapsulate the identifiers of pins on a given \texttt{Component} subtype. It holds only the \texttt{id} and \texttt{name} values and has no logic on its own. The prototypes are parsed from the component configuration file, and are held in the static \texttt{ComponentConfiguration} member of the \texttt{Component} subtype. They are passed to the \texttt{Pin}'s constructor during the completion of \texttt{Component} instances.

\subsection{\texttt{ElectricalNode} class}

The \texttt{ElectricalNode} holds a reference to all of its connected pins in a list. Its digital state is determined by how many of its connected pins are driving. It is low if this value is zero, high otherwise.

The pins of the node are modified using the \texttt{AddPin} and \texttt{RemovePin} methods. Inside these methods, the \texttt{ConnectToNode} and \texttt{Disconnect} methods are called on the pin passed in the argument. The \texttt{DrivingPins} count is updated. While the pin is connected, the \texttt{Propagate} method is either subscribed or unsubscribed from the pin's \textit{StateChangedEvent}.

The \texttt{Propagate} method updates the \texttt{DrivingPins} counter. If the state has been changed by this update \footnotemark, the \texttt{NotifyStateChanged} is called on all pins connected to the node, except the one that triggered the change. 

\footnotetext{I.e. when the counter changed from 0 to 1 or vice versa.}

\subsection{\texttt{Component} abstract class}

An instance of the \texttt{Component} class is identified by both the name of the instance and the name of its type. It holds the \texttt{Transform} (position, rotation and scale within the scene), the currently rendered sprite, and instances of its pins. Reference to a pin is acquired through the \texttt{GetPin} methods, identified either by its name or id. The rendered sprite can be changed with the \texttt{UpdateSprite}, which also identifies sprite by its name, or its reference. These methods invoke the \textit{SpriteChangedEvent}, which is listened to from the \texttt{ComponentControl}. This updates its visuals in the Avalonia UI window.

The user should define the logic of their custom type through the virtual methods. Not all of them need to be implemented, so they cannot be made abstract. The \texttt{OnControlPress} and \texttt{OnControlRelease} are called from the \texttt{ComponentControl}. The \texttt{OnInitialized} is called from the \texttt{SceneFactory} once the component instance has been completed. \texttt{OnPinConnected}, \texttt{OnPinDisconnected} and \texttt{OnPinStateChanged} are subcribed to the appropriate \textit{events} in their pin instances. The semantics and usage of these methods is better described in the subsection \ref{subsec:specs_component_class}.

The \texttt{ComponentConfiguration} structs are stored statically inside the class as a dictionary, with component type name as the key. They store the metadata, but more importantly the pin prototypes from which pin instances are created, and the dictionary of compiled sprites, with the SVG file name as the key (excluding the \texttt{.svg} extension). 

The pins are instantiated in the \texttt{InitPins} method, by passing the \texttt{PinPrototype} to the \texttt{Pin} constructor. Its \textit{events} are subscribed to here.

\subsection{\texttt{SceneFactory} static class}

The \texttt{SceneFactory} class ties the whole loading process together. It takes the path to the scene file and the directory containing the component assets in its constructor, saving them in private fields. It also holds the final \texttt{Scene} instance, which can be retrieved through a public property once the loading is complete.

The main work is done in the \texttt{LoadResources} method, which follows the flow shown in Figure~\ref{fig:cm_project_factory}. It starts by creating a new \texttt{ISceneLoader} to load the scene file, get the list of component types, and instantiate the raw components. It then loops through each component type found in the scene. For each type, it loads additional data from other files, and then configures every individual instance of that type. Finally, it instantiates the electrical nodes, packages them with the components into a new \texttt{Scene} object, and returns it.

The \texttt{CompleteComponentType} method handles the setup for a specific type of component. It first calls \texttt{CreateConfig} to get the \texttt{ComponentConfiguration} of the type. If that succeeds, it uses the \texttt{ISvgLoader} to load all SVG files from that directory. It resizes these SVGs to the size specified in the configuration and compiles them into usable sprites. These sprites are stored back in the configuration, and the finished configuration is registered globally by calling the component's static \texttt{AddConfiguration} method.

The \texttt{CreateConfig} helper method finds and loads the YAML configuration file for a component type. It looks inside the component's directory for a file with the exact same name as the directory, but with a \texttt{.yaml} extension. If the file exists, it uses the \texttt{YamlConfigurationLoader} to load and return the configuration. If the file is missing, it logs an error and returns null, which prevents the loader from trying to instantiate components of this type.

The \texttt{CompleteComponentInstance} method finalizes each individual component instance. It sets the base size of the component's SVGs to match the image size from its \texttt{ComponentConfiguration} and calls the internal pin initialization. It also assigns the first compiled sprite from the configuration as the active sprite, or fallback to an empty \texttt{SKSvg} if no sprites were loaded. At the end, it calls the component's \texttt{OnInitialized} method.

\subsection{SVG Loader}

\texttt{ISvgLoader} exposes functions for loading, modifying and compiling SVG documents. We say documents, because SVG is a readable XML-like format. So the first step is usually loading it into memory as a document object model. In this representation, we can freely change properties of individual picture elements are groups. We keep things simple by only changing the overall size of the picture to the default image size of a component type. After all necessary changes are done, the document is compiled into raw binary data, a blueprint on how the picture should be drawn. This data is wrapped an \texttt{SKSvg} type from \texttt{SkiaSharp}.

\begin{lstlisting}[language=csharp, caption={\texttt{ISvgLoader} interface}]
public interface $ISvgLoader$ {
    public void #LoadDocuments#(string componentPath);
    public void #ResizeDocuments#($Vector2$ desiredSize);
    public $Dictionary$<string, $SvgDocument$> #GetDocuments#();
    public $Dictionary$<string, $SKSvg$> #CompileSvgs#();
}
\end{lstlisting}

Parsing the file and compiling the document is very expensive. A scene with even a few changing SVGs had a considerable lag when using built-in Avalonia, or available Avalonia libraries. That's why we have to compile all the SVGs once before the scene is ready to be used. Multiple component instances can be drawn using the same image data, that's why we store it in the \texttt{ComponentConfiguration} struct, shared between all components of the same type. A single component only holds a reference to the image.

The \texttt{SkiaSvgLoader} uses the \texttt{SvgDocument} type from the \texttt{Svg} NuGet package to create the DOM from a file. On this type, the resizing is done using the \texttt{Width}, \texttt{Height} and \texttt{AspectRatio} properties. The compilation is done using \texttt{Svg.Skia} NuGet package, that extends both \texttt{Svg} and \texttt{SkiaSharp} libraries. The compilation is done using the static \texttt{CreateFromSvgDocument} function on \texttt{SKSvg} class. This is done for all files with the \texttt{.svg} extension in the component directory.


\subsection{Configuration Loader}

The \texttt{IConfigurationLoader} requires only one method, \texttt{LoadConfig}, that creates \texttt{ComponentConfiguration} struct from the given path. The implementation \texttt{YamlConfigurationLoader} utilizes the \texttt{YamlDotNet} NuGet package. It implements a deserializer, that reads the YAML from a file and saved is it to memory as structure with equivalent fields and data types definitions. For that we define the \texttt{ConfigDto} struct \footnotemark. 

\footnotetext{The DTO meaning data transfer object.}

\begin{lstlisting}[language=csharp, caption={\texttt{YamlConfigurationLoader.ConfigDto} struct}]
public record struct $ConfigDto$() {
    public required string Name;
    public string Description = "";
    public string ImageSize = "";
    public uint Pins = 0;
    public $List$<string> PinNames = [];
}
\end{lstlisting}

It serves as an intermediate step before we shape the loaded data into the form we need. The rest is quite simple. \texttt{Name} and \texttt{Description} have exactly the same. The \texttt{ImageSize} size arrived as a string with two numbers seperated by space, so we just split them and parse them separately. \footnotemark The \texttt{Pins} are created as prototypes, where the unnamed prototypes have IDs ranging from zero to to the value specified in the DTO and no name. The named prototypes have names from the DTO and IDs in the same order as were they names listed in the DTO. The \texttt{Sprites} dictionary is not yet populated, as we want to avoid a dependency on another loader.

\footnotetext{This is a non-standard YAML notation of 2D vectors. We created the static \texttt{YamlUtils} class to handle these vectors. See the specification at \ref{yaml_notation}}

\begin{lstlisting}[language=csharp, caption={\texttt{YamlConfigurationLoader.CreateConfig} method}]
private $ComponentConfiguration$ #CreateConfig#(
    $ConfigDto$ configDto, string configPath) => new() {
        Name = configDto.Name,
        ComponentPath = $Path$.#GetDirectoryName#(configPath)!,
        Description = configDto.Description,
        ImageSize = $YamlUtils$.#StringToVector2#(configDto.ImageSize) 
            ?? $ComponentConfiguration$.DEFAULT_IMAGE_SIZE,
        Pins = #CreatePins#(configDto)
    };
\end{lstlisting}

\subsection{Scene Loader}

The \texttt{ISceneLoader} is responsible for parsing the entire scene. That means creating both \texttt{Component} and \texttt{ElectricalNode} instances, and register all component types used in the scene. The component instances are not yet complete, their pins and sprite require data from the other loaders. That's why we need na effective way to group them by their type.

The \texttt{YamlSceneLoader} also uses the \texttt{YamlDotNet} package and its deserializer. The idea of the DTO also stays the same, just its definition is more complicated due to all the nested types. The \texttt{LoadFile} method does nothing more than deserializing the path into the DTO. The actual creation of the instances is done in \texttt{InstantiateComponents} and \texttt{InstantiateNodes}, which must be called in that order for the operation to make sense.

\begin{lstlisting}[language=csharp, caption={\texttt{YamlSceneLoader.SceneDto} definitions}]

public class $TypesMap$ : $Dictionary$<string, $ComponentsMap$>;
public class $NodesList$ : $List$<$List$<string>>;

public record $SceneDto$() {
    public required $TypesMap$ Components { get; init; } = [];
    public required $NodesList$ Nodes { get; init; } = [];
}

public record $ComponentDto$() {
    public string Position = "0 0";
    public float Rotation = 0;
    public string Scale = "1 1";

}
\end{lstlisting}

\subsubsection{Instantiating components}

\texttt{InstantiateComponents} loops through the nested dictionary of types and DTOs inside the deserialized scene. For each type name encountered, it attempts to resolve the C\# type of the currently executing assembly (if not already cached) and then loops through individual component DTOs to kick off their creation.

The \texttt{CreateComponentInstance} method uses reflection to dynamically instantiate the object using its type name. Once the instance exists, it assigns it the information from \texttt{ComponentDto} (i.e. \texttt{Transform} and custom properties) and registers the finished instance into both the type-grouped and name-grouped lookup dictionaries.

The custom properties are processed in the \texttt{HandleExtraProps} method from the \texttt{ExtraProps} member of the \texttt{ComponentDto}. User-defined properties are not part of the base component layout. They are acquired through reflection by matching properties of the concrete component class against the dictionary keys. If a property exists, the input type is converted dynamically to the destination property type, and the value is updated.

\begin{lstlisting}[language=csharp, caption={\texttt{YamlSceneLoader.CreateComponentInstance} method}]
private void #CreateComponentInstance#(
    string typeName, 
    string componentName, 
    $ComponentDto$ componentDto) 
{
    try {
        $Type$ type = _typesByName[typeName];
        $Component$ compInstance = 
            ($Activator$.#CreateInstance#(type, typeName) 
                as $Component$)!;

        compInstance.Name = componentName;
        compInstance.Transform = #ParseTransform#(componentDto);
        #HandleExtraProps#(compInstance, componentDto.ExtraProps);

        _instancesByType[typeName].#Add#(compInstance);
        _instancesByName[componentName] = compInstance;
        $ComponentManager$.#LogInfo#("...");
    }
    catch {
        $ComponentManager$.#LogError#("...");
    }
}
\end{lstlisting}

\subsubsection{Instantiating nodes}

After all components exist, the \texttt{InstantiateNodes} iterates through the list of node arrays, creating a new \texttt{ElectricalNode} container for each. It splits the string representations (e.g. "Component.Pin") to locate the parent instance by name, resolves the pin using either its numeric index or name identifier, and adds \texttt{Pin} references to the node.

\section{Main project with Avalonia UI}

The \texttt{Main} project has been imlemented with the MVVM architecture in mind. The only model of the application is the \texttt{Component} class from the \texttt{Component Management} project (see \ref{sec:cm_project_circuitry_and_scenes}).

The views and viewmodel files are put in separate directories. Views consist of \texttt{.axaml} files and the codebehind \texttt{.axaml.cs} files. These codebehind files contain a partial class with a name of the the view, inheriting from Avalonia's \texttt{UserControl} class. These can be used to write logic for the views as well, but we delibarately left them empty. The only exception is the \texttt{App.axaml.cs}, which has to call the \texttt{MainController} once the UI is initialized.

The \texttt{.axaml} files define the visual tree of the application. The root of this application is \texttt{App.axaml}, which owns the main window and includes some default styles from \texttt{Styles/General.axaml}. Our application doesn't create any other windows, everything is situated in the \texttt{MainWindow.axaml} and its controls.

The \texttt{MainController} intializes the majority of the application. It invokes the \texttt{SceneFactory} and populates the \texttt{CircuitCanvas} viewmodel with the parsed scene. Then it chooses the IPC implementation based on the command line arguments, and the passes the IPC instance to the \texttt{Arduino} component.

\begin{lstlisting}[language=csharp, caption={\texttt{MainController.InitScene} function}]

\end{lstlisting}

The scene is created by the \texttt{SceneFactory} from the files specified in the command line argument. Then the \texttt{CircuitCanvas} viewmodel is created, and its \texttt{Components} property is populated with the components from the scene. Then the viewmodel is set to be the data context for the \texttt{CircuitCanvas} view. But what does setting a data context mean?

As we mentioned at the beginning of this section, all the views have a codebehind with a partial class named after the view. The other parts of this class are generated during compilation of the \texttt{.axaml} file. The XAML knows the name of the partial class it should generate thanks to the mandatory \texttt{x:Class} attribute in the root element of the control.

We never explicitly defined a \texttt{CircuitCanvas} property in the \texttt{MainWindow} codebehind. Instead we added the attribute \texttt{x:Name="CircuitCanvas"} to the concrete \texttt{CircuitCanvas} XAML element. This tells the XAML compiler to generate the property with the given name in the \texttt{MainWindow} class, and populate it automatically with the concrete instance during Avalonia initialization. That is with all the properties set either through the codebehind, or through the attributes on the XAML element itself. \cite{avalonia_name_generator}

\texttt{DataContext} is a property inherent to all the controls. It can be an instance of any one class, with which the control exchanges data through binding. That means, the control can change properties automatically when the values of the binded property changes. And in the other direction, the property of the data context automatically updates when a user interacts with the control. \footnotemark 

In MVVM, data contexts are the best way to bind views to their viewmodels. \cite{avalonia_data_context} The type of the \texttt{DataContext} must be set to match the type of the view model in XAML. This can be done in more than one way. We do it by adding the \texttt{x:DataType} attribute to the root element of \texttt{CircuitCanvas.axaml}. The code below shows a simplified XAML example of how these two steps have been done.

\xxx{syntax highlight}

\subsubsection{Arduino initialization}

First we try to find the \texttt{Arduino} component in the scene. It is perfectly valid to have a scene without Arduino, so in that case we only throw a warning and skip the rest of the initilazition. With the way this is handled, only one Arduino can reasonably work in a scene. A better implementation, would be to initialize the Arduino with IPC inside the component itself, and get the IPC arguments like port or shared memory name from extra properties in the scene file. That way, it would be possible to have multiple Arduino components in the scene, each connected to its own simulator. But the approach has not been properly tested. Since the IPC arguments are at the moment passed through the command line, we can only have one Arduino anyway.

Subsequently the IPC logger, the text encoder and the IPC adapter is initialized. The \texttt{IIpcAdapter} implementation is chosen based on the \texttt{IpcType} option from the command line arguments. If the chosen type is \texttt{None}, the function is skipped completely. After instantiating the adapter, we perodically try to connect to it in a new task, and immediately pass it to the \texttt{Arduino} component. Once the adapter connects (if ever), the component will be able to react to the incoming events.

\subsection{\texttt{ComponentControl} custom control}

\xxx{Adjust Alt key behavior}

The \texttt{ComponentControl} wires up the elements of the circuit canvas to the \texttt{Component} class in the \texttt{Component Management} project. The class inherites from \texttt{Control}, in order to override its \texttt{Render} method, and respond to Avalonia \textit{events}. The \texttt{InvalidateVisual} method is called when the sprite of the \texttt{Component} changes. It notifies the Avalonia renderer, that it is time to redraw the control. The \texttt{Render} method is then called, using the new sprite.

\texttt{OnPointerPressed} and \texttt{OnPointerReleased} are tied to the components \texttt{OnControlPress} and \texttt{OnControlRelease} methods. These are called when the user clicks the control on the canvas with their mouse. \texttt{OnAttachedToVisualTree} and \texttt{OnDetachedFromVisualTree} are used here, to subcribe to the global Alt key \textit{event}. The Alt key modifies whether should the control be still pressed, when the pointer is released.

We need to specify which \texttt{Component} instance from \texttt{CircuitCanvas} view is tied to this control. I.e. we need to define a property, we can set up from the canvas' XAML, that we can then access from the control. Unfortunately, Avalonia is very obtuse with its property system, and requires quite a lot of boilerplate.

First we define the \texttt{\_component} backing field. Then we define a static \texttt{DirectProperty} field, intialized by the \texttt{RegisterDirect} method, templated by the type of the control and type of the backing field. Then we define a regular property with getter and setter. The setter must contain the \texttt{SetAndRaise} in order to notify the control when its value changes. The property is passed to \texttt{RegisterDirect} and \texttt{DirectProperty} is passed to \texttt{SetAndRaise}. Then, once the value of the property changes through XAML binding, the \texttt{OnPropertyChanged} method is called. From the argument of this call we determine the property changed was our \texttt{Component}, and we can subscribe or unsubscribe from its \textit{SpriteChangedEvent}.

\subsection{\texttt{SvgDrawOp} drawing operation}

The \texttt{SvgDrawOp} is used within the \texttt{Render} method of \texttt{ComponentControl}. It takes the compiled SVG of the \texttt{Component} as an argument, as well as its \texttt{Transform} converted into a transformation matrix. This conversion is done through an extension method within the \texttt{GeometryExtensions} class. The method creates elementary 3x3 translation, rotation and scale matrices and multiplies them together. \texttt{SKMatrix}, the type of the transformation matrix, comes from the \texttt{SkiaSharp} library. The same one that defines the SVG type, and many other systems used by the Avalonia's rendering pipeline.
\footnote{\texttt{SkiaSharp} is the C\# API of \texttt{Skia}, an open source 2D graphics engine. It is owned by Google and is used in some of their products like Chrome or Android for rendering PDFs, shaders and vector graphics. \cite{skia} The API is implemented in many more popular languages. }

\subsubsection{Hit testing}

\begin{lstlisting}[language=csharp, caption={\texttt{SvgDrawOp.HitTest} method}]

This method is called during pointer \textit{events}, e.g. when the user clicks on the canvas. It checks whether or not is the pointer located on top of the control. We use this in order to trigger the \texttt{OnPointerPressed} and \texttt{OnPointerReleased} handlers of the \texttt{ComponentControl}. 

The argument \texttt{p} is the location of the pointer relative to the circuit canvas. The \texttt{\_hitBox} is given by the SVG. It is the rectangle in which the picure is inscribed. We described its coordinates in the local space of the picture. The rectangle's left corner is located in the origin of the local space, and its dimensions are given by \texttt{ImageSize} property we specified in the component's configuration. Using the inverse of the tranformation matrix, we can map \texttt{p} from the space of the canvas, to the picture's local space. If it lies within the \texttt{\_hitBox} area, we consider it a hit.

\subsubsection{Rendering}

\begin{lstlisting}[language=csharp, caption={\texttt{SvgDrawOp.Render} method}]

    canvas.#Concat#(ref Transform);
    canvas.#DrawPicture#(Svg.Picture);
    canvas.#Restore#();

\end{lstlisting}

The method takes \texttt{ImmediateDrawingContext} as an argument, which is passed down from the argument of the \texttt{Render} method of \texttt{Control}. It defines the environment in which the picture is going to be drawn. We need to access the API of the Skia's canvas, upon which the Avalonia's drawing contexts are built and render their own vector graphics. We use it to draw our own \texttt{SKSvg} sprite. We do that by obtaining the SkiaSharp API lease from the context's features. \footnotemark

The Skia canvas coordinates are already in the vector space of the parent control (the \texttt{CircuitCanvas}). So we can just multiply the tranformation matrix of the canvas with the transformation matrix of the control, in order to place it on the canvas with the right translation, rotation and scale. \section{IPC Adapter project} \label{sec:f_ipcadapter}

\begin{figure}[htbp]
    \centering
    \includegraphics[width=\textwidth,keepaspectratio]{./img/uml/f_IPC.pdf}
    \caption{\texttt{IpcAdapter} project architecture}
    \label{fig:ipc_project}
\end{figure}

This project attempts to unify the behavior of the TCP adapter and the shared memory adapter in a similar manner the \texttt{ipcAdapter.hpp} did in the backend. \texttt{IIpcAdapter} is initialized by the \texttt{MainController} and the implementation is chosen based on command line arguments. It is then passed to the \texttt{Arduino} component to handle.

This time around, the adapter is not dependant on a \texttt{Simulator}-like entity to pass the events to. It simply invokes the \textit{ReceivedEventEvent}, which passes the event to any subscribed handler, i.e. the \texttt{Arduino} class. The actions of the \texttt{Event} struct are also not defined by static methods, but determined in the \texttt{Arduino} class itself.

\subsection{\texttt{Event} struct}

\begin{lstlisting}[language=csharp, caption={\texttt{Event} struct}]
public enum $EventType$ { Unknown, Write, PinMode, Reboot }

public struct $Event$ {
    public const int MAX_ARGS = 3;
    
    public $EventType$ Type;

We've seen this structure many times before. In contrast with its backend counterpart, it doesn't include the \texttt{action} lambda, the factory methods, nor any other helper functions.

\subsection{\texttt{IIpcAdapter} interface}

\begin{lstlisting}[language=csharp, caption={\texttt{IIpcAdapter} interface}]
public interface $IIpcAdapter$ {
    private const int RECONNECT_DELAY = 3000;
    public bool IsConnected { get; }
    public event $Action$<$Event$> ReceivedEventEvent;
    
    public void #SendEvent#($Event$ @event);
    
    public $Task$<bool> #TryConnectAsync#();
    public void #Disconnect#();

    public async $Task$ #ConnectAsync#() { 
        while (!await #TryConnectAsync#()) {
            await $Task$.#Delay#(RECONNECT_DELAY);
        }
    }
    
    public void #Reconnect#() {
        #Disconnect#();
        $Task$.#Run#(#ConnectAsync#);
    }

\end{lstlisting}

The \texttt{SendEvent} method is similar to the \texttt{on\_event} method of the \texttt{ipcAdapter.hpp}. It is also used to send an event over IPC, by first encoding it to a message, just the dependencies are now inversed. The adapter isn't dependant on the simulator but vice versa. So we don't need to pass it anywhere, just call it directly from a component. And the events are not directly applied by calling the component, but by invoking the \textit{ReceivedEventEvent} that the components subscribe to.

The interface also offers some more methods for handling the connection. \texttt{TryConnectAsync} and \texttt{Disconnect} must be implemented by the concrete adapters as they are dependant on the technology used. We want them exposed, because in the future, it could be desirable to change the IPC setup during runtime (for example, connecting to a simulator on a different port).

\texttt{ConnectAsync} tries to periodacally connect to the every three seconds. Useful when the simulator isn't running yet. It establishes connection automatically when it is ready. \texttt{Reconnect} is useful, when there was some kind of sudden failure.

\subsection{\texttt{TextEncoder} class}

The \texttt{IEncoder} and \texttt{TextEncoder} are nearly equivalent to \texttt{encoder.hpp} and \texttt{textEncoder.cpp} in the backend. The interface still exposes methods for encoding and decoding, where decoding return a success state, and outputs the event to a parameter.

The implementaion defines constants for formats of individual parts, and it still uses string formatting to fill them with event's arguments. Just the decoding has become much more convenient thanks to .NET's built-in functions.

\subsection{Arduino component}

The configuration of the component defines 14 digital pins (\texttt{D0} up to \texttt{D13}) and 6 analog pins (\texttt{A0} up to \texttt{A5}). It has exactly one sprite, simply named \texttt{Arduino}. The partial class \texttt{Arduino} is split into two files \texttt{Arduino.cs} and \texttt{Arduino.Events.cs}. This is to establish seperation of concerns while still having access to all the private methods of both the side handling pin interaction and the side handling consumption and creation of events.

\subsubsection{\texttt{Arduino.cs}}

The front end side of the class defines a method for connecting to the IPC and some helper functions for conversion between pin indices and pin instances. The rest of the methods each reflects one of the possible IPC events. \texttt{WriteToPin} sets a value on a digital pin,\texttt{SetPinMode} sets a pin to write only or read only mode, and \texttt{Reboot} defaults all pins to a low state.

\begin{lstlisting}[language=csharp, caption={\texttt{Arduino.ConnectToSimulator} method}]
public void #ConnectToSimulator#($IIpcAdapter$ ipc) {
    _ipc = ipc;
    
    _ipc.ReceivedEventEvent += e => {
        _uiContext.#Post#(_ => #ProcessRemoteEvent#(e), null);    
    };
}
\end{lstlisting}

The class gets the \texttt{IIpcAdapter} object from the \texttt{MainController} class of the \texttt{Main} project. It is already connected to the IPC, or trying to connect periodically. We have to subscribe to its \textit{ReceivedEventEvent}, which passes the decoded event as an argument.

We have to be careful about the thread from which we consume the event. Most UI frameworks are single-threaded, and don't handle well, if more than one thread tries to enter the rendering pipeline. Consuming an event means we will have to at some point change a sprite of component connected to Arduino, and thus invalidate a visual of its associated \texttt{ComponentControl} in an Avalonia view. But the IPC listening system most likely executes on a different thread than the UI. Because of this the application kept freezing almost immediately after launch of the IPC, probably due to some deadlock. 

We solved it by wrapping the \texttt{ProcessRemoteEvent} in \texttt{\_uiContext.Post} method. \texttt{\_uiContext} is an instance of the .NET's \texttt{SynchronizationContext} class. It represents the execution context of a single thread. We set the instance by calling \texttt{SynchronizationContext.Current} while constructing the component in the UI thread (the one that executes \texttt{MainController} functions). The \texttt{Post} method is a non-blocking method that schedules a delegate to be executed on the given thread. So the \textit{ReceivedEventEvent} is subscribed to a lambda, which schedules the consumption of the event on the correct thread.

\begin{lstlisting}[language=csharp, caption={\texttt{Arduino.OnPinStateChanged} method}]
public override void #OnPinStateChanged#($Pin$ pin) {
    int idx = #GetPinIndex#(pin.Name!);
    
    if (_ipc?.IsConnected ?? false) {
        #InvokeWriteEvent#(idx, pin.State);
    }
}
\end{lstlisting}

The rest of the logic is simple. If we detect a change on one of the input pins, we check that IPC is connected. If it is, we create the appropriate event with the appropiate arguements.
 
\subsubsection{\texttt{Arduino.Events.cs}}

The \texttt{ProcessRemoteEvent} method comes from the event side of the \texttt{Arduino} class. It calls one of the \texttt{WriteToPin}, \texttt{SetPinMode} or \texttt{Reboot} methods based on the type of the event. Then it stores the LVT of the event into the \texttt{\_lastSeenLvt} member.

\begin{lstlisting}[language=csharp, caption={Sending the Write event with \texttt{Arduino.Events.cs}}]

    new() {
        Type = $EventType$::Write,
        Args = [ pinId, (int)value ]
    };

    #SendLocalEvent#(#CreateWriteEvent#(pinId, value));

    @event.LocalVirtualTime = _lastSeenLvt;
    _ipc?.#SendEvent#(@event);
}
\end{lstlisting}

When the component needs to produce an event, we call one of the \texttt{Invoke*} methods. It creates the \texttt{Event} struct with the specific type and arguments and sends it via IPC with the \texttt{\_lastSeenLvt}. It isn't necessary to assign a unique LVT to different events. They will always be sent to the backend in the same order. All that matters is that a certain set of local events occurred between 2 remote events produced by the backend.

\subsection{\texttt{TcpClient} class}

The TCP Adapter is built on the .NET's native TCP Client implementation. It uses the \texttt{NetworkStream}, through which the actual data is sent. Our own \texttt{TcpClient} class wraps around the client. It establishes connection on the given port and initializes loops for sending and receiving data in new tasks. The received chunks of data are passed to the \texttt{TcpMessageHandler}, which divides them into individual messages. Messages to be sent are passed over to the \texttt{TcpClient} thorugh the handler as well. \texttt{TcpMessageHandler} implements the \texttt{IIpcAdapter}.

The \texttt{TcpClient} is constructed with an IP address and a port (i.e. localhost and the port of the backend). Only in the \texttt{TryConnectAsync} method is the underlying \texttt{\_client} created and connected to asynchronously. Its \texttt{NetworkStream} is acquired. The \texttt{LoopReadAsync} and \texttt{LoopWriteAsync} methods are then run in seperate tasks. The cancellation token (\texttt{\_cts}) is created. The token can be used to terminate these loops, once a disconnect or an error occurs. The \texttt{Disconnect} method cancels this token, and disposes of both the stream and the client.

\subsubsection{Receiving messages}

\begin{lstlisting}[language=csharp, caption={\texttt{TcpClient.LoopReadAsync} method}]
private async $Task$ #LoopReadAsync#() {
    try {
        while (!_cts.IsCancellationRequested) {
            int bytesRead = await _stream!.#ReadAsync#(
                _buffer, 0, _buffer.Length, _cts.Token);
            if (bytesRead == 0) {
                ReachedEof?.#Invoke#();
                return;
            }

            _logger?.#LogDebug#("...");

            var byteChunk = $Encoding$.ASCII.#GetString#(
                _buffer, 0, bytesRead);
            ReadBytesEvent?.#Invoke#(byteChunk);
        }
    }
    catch ($Exception$ e) {
        _logger?.#LogError#("...");
    }
}
\end{lstlisting}

The read loop is conditioned by the \texttt{\_cts} token. Inside, the stream's \texttt{ReadAsync} method is used to read any new available data on the stream, and copy it to the \texttt{\_buffer}. \texttt{\_buffer} is a simple array with size of 4096 bytes. We could choose any other size, but this one gives us a lot of breathing room if the stream happens to get clogged by many messages. The method blocks until new data is available.

By passing 0 to the 2nd argument of the method, new data will always be copied to the start of the buffer, overwriting any old data. By passing the \texttt{\_cts} token as argument, the method will throw an \texttt{OperationCancelledException} once the token has been canceled. The method returns the number of bytes it has successfully read. If this number is 0, it means the connection has been gracefully closed. \cite{msdocs_network_stream}. In that case we invoke the \textit{ReachedEof event}.

Since the new data is always at the start of the buffer, we can pass it to the \texttt{Encoding.ASCII.GetString} function in order to convert it into a string of ASCII characters. The number of bytes we want to convert is the value \texttt{bytesRead} returned from \texttt{ReadAsync}. We pass the converted bytes to the \textit{ReadBytesEvent}, to which the \texttt{TcpMessageHandler} subscribes to. There the string can (but doesn't have to) be pieced together into a list of meaningful messages.

\subsubsection{Sending messages}

\begin{lstlisting}[language=csharp, caption={\texttt{TcpClient.SendMessageAsync} method}]
    public async $Task$ #SendMessageAsync#(string message) {
        await _sendQueue.Writer.#WriteAsync#(message);
    }
\end{lstlisting}

For sending messages we utilize the .NET's \texttt{Channel} class. This a thread safe implementation of the concurrent producer/consumer pattern. It is a FIFO data structure, so we call the instance \texttt{\_sendQueue} in our code. The messages are added to the channel through the \texttt{Writer} property and removed through the \texttt{Reader} property. The channel is unbounded, meaning that it doesn't block the writing thread, if the number of items reaches a certain limit. 

\begin{lstlisting}[language=csharp, caption={\texttt{TcpClient.LoopWriteAsync} method}]
private async $Task$ #LoopWriteAsync#() {
    try {
        while (await _sendQueue.Reader.#WaitToReadAsync#()) {
            while (_sendQueue.Reader.#TryRead#(out var message)) {
                var data = $Encoding$::ASCII.#GetBytes#(message);
                await _stream!.#WriteAsync#(data, 0, data.Length);
                _logger?.#LogDebug#("Sent {data.Length} bytes...");
            }
        }
    }
    catch ($Exception$ e) {
        _logger?.#LogError#("TCP Writing loop failed: {e.Message}");
    }
}
\end{lstlisting}

The write loop reads from the \texttt{\_sendQueue}. It blocks the thread until there is at least one item available to be read. If the channel has been invalidated, the method returns false. In the inner loop, all available string messages are read and converted to ASCII bytes. The bytes are then copied asynchonously into the network stream.

\subsection{\texttt{TcpMessageHandler} class}

The \texttt{TcpMessageHandler} implements the \texttt{IIpcAdapter} and owns our \texttt{TcpClient} wrapper. It let's the client handle the \texttt{TryConnect} and \texttt{Disconnect} interface method by calling the client's methods with the same name. The \texttt{SendEvent} method is implemented by encoding the event, appending the delimiter, and passing it to the client's \texttt{SendMessageAsync} method in a new task. When the client's \textit{ReachedEof} is invoked, the handler tries to reconnect using the method inherited from the interface.

\subsubsection{Handling incoming messages}

\begin{lstlisting}[language=csharp, caption={\texttt{TcpClient.OnBytesRead} method}]
private void #OnBytesRead#(string bytechunk) {
    foreach (char c in bytechunk) {
        if (c == Delimeter) {
            #HandleMessage#(_messageBuilder.#ToString#());
            _messageBuilder.#Clear#();
        }
        else {
            _messageBuilder.#Append#(c);
        }
    }
}
\end{lstlisting}

The handler's \texttt{OnBytesRead} method subscribes to the client's \textit{ReadBytesEvent}. It receives a string that may contain the entire message, multiple messages, or just their pieces. So we add any incoming characters to the \texttt{\_messageBuilder} and look for the delimeter. Once a whole message has been put together, it is passed to the \texttt{HandleMessage} method, and the builder is cleared.

In \texttt{HandleMessage}, the message is decoded into an \texttt{Event} struct and the \textit{ReceivedEventEvent} is invoked.

\section{Shared Memory Adapter project}

\begin{figure}[htbp]
    \centering

    \caption{\texttt{SharedMemoryAdapter} project architecture}
    \label{fig:shmem_project}

The project is structured in a very similar way as its backend counterpart. It is implemented using the .NET's \texttt{MemoryMappedFile} class. Despite the name, it is the intended way to open the shared memory \cite{msdocs_memory_mapped_files}. We do that by passing it the virtual file generated by the \texttt{boost::interprocess} library in the backend. Portions of the memory can be isolated and accessed via \texttt{MemoryMappedViewAccessor} which serves a similar purpose as the \texttt{mapped\_region} type in \texttt{boost::interprocess}.

The \texttt{TwoWayBuffer} implementing the \texttt{IIpcAdapter} now plays the role of both the \texttt{TwoWayBuffer} and \texttt{shmemAdapter.cpp} in the backend. It opens the \texttt{MemoryMappedFile}, listens for messages, encodes events and passes them to its circular buffers. The \texttt{CircularBuffer} implementation is almost identical to the in the backend, with just a few caveats of the change in language. This time around, there is no \texttt{MemoryRegion} equivalent, as all the useful abstractions have already been built in to the \texttt{MemoryMappedViewAccessor}.

\subsection{\texttt{CircularBuffer} class}

The class owns the \texttt{MemoryMappedViewAccessor}. It still offers the \texttt{Write} and \texttt{ConsumeUntil} methods. We didn't define a dedicated iterator for the buffer so we need to implement finding the delimeter manually. The \texttt{Write} method is little less flexible, as it only accepts string and spans of bytes. In the backend, we converted any type into a span of bytes anyways, so the body of the method stays basically the same. Just instead of the \texttt{memcpy} function with pointers we use the accessor's \texttt{ReadArray} and \texttt{WriteArray} methods addressed directly with offsets. For full context see \ref{subsec:b_circular_buffer}

\subsection{\texttt{TwoWayBuffer} class}

\texttt{TwoWayBuffer} is also very similar to how the same class and the adapter were implemented in the back. It still opens and owns shared memory in the form of \texttt{MemoryMappedFile} object. It also owns and accesses both the consuming and the producing \texttt{CircularBuffer}. The \texttt{SendEvent} and \texttt{ListenForRemoteEvents} methods are equivalent to the \texttt{on\_event} and \texttt{run\_shmem\_consumer} in \texttt{shmemAdapter.cpp} (see \ref{subsec:shmemadapter}).

Only acquiring the shared memory object needs a little more work than before. Providing the object's name is not enough. We need to locate the virtual file created by the backend. This happens in the \texttt{OpenSharedMemory} method. The location changes drastically depending on the host's operating system. For specifics, see the subsection \ref{subsec:shmem_specs}.
